## 1. 문서 작성 개요

### 1.1 목적
이 가이드는 **Girls Band Tabi** 프로젝트의 모든 기술 문서가 일관성 있고 관리 가능한 형태로 작성되도록 하는 것을 목적으로 합니다.

### 1.2 적용 범위
- 개발 가이드 문서
- API 문서
- 아키텍처 문서
- 보안 가이드
- 운영 매뉴얼
- 기타 기술 문서

### 1.3 작성 원칙
- **명확성**: 누구나 이해할 수 있는 명확한 언어 사용
- **일관성**: 동일한 형식과 스타일 적용
- **최신성**: 정기적인 업데이트를 통한 최신 정보 유지
- **접근성**: 쉽게 찾고 활용할 수 있는 구조

## 2. 버전 관리 시스템

### 2.1 버전 표기 형식
모든 문서는 **Semantic Versioning(시맨틱 버저닝)** 형식을 따릅니다:

```
v[Major].[Minor].[Patch]
```

#### 버전 증가 규칙
- **Major(메이저) (v2.0.0)**: 문서 구조나 내용의 대대적인 변경
- **Minor(마이너) (v1.1.0)**: 새로운 섹션 추가 또는 중요한 내용 수정
- **Patch(패치) (v1.0.1)**: 오타 수정, 링크 업데이트, 소규모 개선

### 2.2 버전 표기 위치
문서 제목과 헤더에 다음과 같이 표기:

```markdown
# 문서명 v1.0.0 📊

**최신 업데이트**: YYYY년 MM월 DD일  
**버전**: v1.0.0  
**상태**: ✅ 최신 | 🟡 업데이트 필요 | ❌ 구버전  
```

### 2.3 초기 버전
- 모든 신규 문서는 **v1.0.0**으로 시작
- 파일명에도 버전 정보 포함

## 4. 파일명 규칙

### 4.1 파일명 형식
```
[문서종류]_v[버전].md
```

### 4.2 문서종류 표준명
- `개발자가이드`: 종합 개발 가이드
- `API문서`: API 레퍼런스(Reference)
- `데이터베이스설계`: DB 스키마(Schema) 및 설계
- `시스템아키텍처`: 시스템 구조 문서
- `보안가이드`: 보안 정책 및 가이드
- `성능최적화가이드`: 성능 개선 가이드
- `코드표준가이드`: 코딩 표준
- `개발환경설정가이드`: 환경 설정 가이드
- `문서작성가이드`: 문서 작성 규칙 (본 문서)

### 4.3 파일명 예시
```
개발자가이드_v1.0.0.md
API문서_v1.2.1.md
데이터베이스설계_v2.0.0.md
```

## 5. 내용 작성 가이드라인

### 5.1 언어 및 표기
- **주언어**: 한국어
- **기술용어**: 영어 용어와 한국어 병기 (첫 등장시)
- **날짜 형식**: YYYY년 MM월 DD일
- **시간 형식**: HH:mm:ss (24시간제)

### 5.2 톤앤매너
- **친근하면서 전문적**: 개발자 친화적 언어 사용
- **명확하고 간결**: 불필요한 수식어 피하기
- **실용적**: 실제 사용 가능한 예제 포함
- **최신성**: 현재 기술 스택과 일치하는 내용

### 5.3 코드 예제 작성
```kotlin
// ✅ 좋은 예제: 명확한 주석과 실제 사용 가능한 코드
@RestController
@RequestMapping("/api/v1")
class UserController(
    private val userService: UserService
) {
    // EN: User information retrieval API
    // KO: 사용자 정보 조회 API
    @GetMapping("/users/{id}")
    fun getUser(@PathVariable id: Long): UserDto {
        return userService.findById(id)
    }
}
```

## 6. 마크다운 스타일 가이드

### 6.1 헤딩(Heading) 구조
```markdown
# 메인 제목 (H1) - 문서당 1개
## 주요 섹션 (H2)
### 세부 항목 (H3)
#### 상세 설명 (H4)
```

### 6.2 강조 및 포맷팅(Formatting)
- **굵게**: `**중요한 내용**`
- *기울임*: `*강조*`
- `인라인 코드`: `` `코드` ``
- 🎯 **이모지(Emoji) 사용**: 제목과 중요 섹션에 적절히 활용

### 6.3 리스트(List) 형식
```markdown
#### 순서가 있는 리스트(Ordered List)
1. 첫 번째 항목
2. 두 번째 항목

#### 순서가 없는 리스트(Unordered List)
- 주요 항목
  - 하위 항목
  - 하위 항목
- 주요 항목
```

### 6.4 표(Table) 형식
```markdown
| 항목 | 설명 | 예시 |
|------|------|------|
| API | REST API | `/api/v1/users` |
| DB | PostgreSQL | 15+ PostGIS |
```

### 6.5 코드 블록(Code Block)
```markdown
```kotlin
// Kotlin 코드 예제
class UserService {
    fun findById(id: Long): User {
        return userRepository.findById(id)
    }
}
```

### 6.6 링크(Link) 및 참조
```markdown
- **내부 링크**: [섹션명](#앵커-이름)
- **외부 링크**: [Spring Boot](https://spring.io/projects/spring-boot)
- **파일 링크**: [설정 가이드](./개발환경설정가이드_v1.0.0.md)
```

## 7. 버전 업데이트 절차

### 7.1 업데이트 트리거
다음 상황에서 버전 업데이트를 수행:

#### Major 업데이트 (v2.0.0)
- 기술 스택(Tech Stack) 대변경 (Spring Boot 버전 메이저 업그레이드 등)
- 아키텍처 전면 개편
- 문서 구조의 근본적 변경

#### Minor 업데이트 (v1.1.0)
- 새로운 기능 추가
- 새로운 섹션 추가
- 중요한 정책 변경

#### Patch 업데이트 (v1.0.1)
- 오타 수정
- 링크 업데이트
- 소규모 내용 개선
- 예제 코드 수정

### 7.2 업데이트 절차

#### 1단계: 변경사항 식별
```bash
# 변경된 내용 확인
git diff HEAD~1 docs/dev/
```

#### 2단계: 버전 결정
- 변경 규모에 따라 Major/Minor/Patch 결정
- 새로운 버전 번호 산정

#### 3단계: 문서 업데이트
```markdown
# 파일명 변경 (필요시)
mv 문서명_v1.0.0.md 문서명_v1.1.0.md

# 헤더 정보 업데이트
**최신 업데이트**: 2025년 11월 13일  
**버전**: v1.1.0  
```

#### 4단계: 변경 로그(Changelog) 작성
각 문서 하단에 변경 이력 추가:
```markdown
## 변경 이력
- **v1.1.0** (2025.11.13): 새로운 보안 정책 섹션 추가
- **v1.0.1** (2025.11.10): 코드 예제 오타 수정
- **v1.0.0** (2025.11.01): 초기 버전 작성
```

## 8. 문서 검토 및 승인

### 8.1 검토 프로세스
모든 문서는 다음 단계를 거쳐 승인:

#### 1단계: 자체 검토
- [ ] 맞춤법 및 문법 검사
- [ ] 링크 유효성 확인
- [ ] 코드 예제 테스트
- [ ] 구조 및 일관성 검토

#### 2단계: 동료 검토
- [ ] 기술적 정확성 검증
- [ ] 가독성 평가
- [ ] 실용성 확인

#### 3단계: 팀 승인
- [ ] 팀 리더 또는 시니어 개발자 승인
- [ ] 최종 버전 확정

### 8.2 검토 체크리스트(Checklist)

#### 내용 품질
- [ ] 기술적으로 정확한가?
- [ ] 최신 정보를 반영하고 있는가?
- [ ] 실제로 따라할 수 있는 내용인가?
- [ ] 대상 독자에게 적합한가?

#### 구조 및 형식
- [ ] 헤더 형식이 올바른가?
- [ ] 버전 정보가 정확한가?
- [ ] 목차가 완전한가?
- [ ] Markdown 문법이 올바른가?

#### 유지보수성
- [ ] 업데이트하기 쉬운 구조인가?
- [ ] 관련 문서들과 일관성이 있는가?
- [ ] 링크가 모두 유효한가?
