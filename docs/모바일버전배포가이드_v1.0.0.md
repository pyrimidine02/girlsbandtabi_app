## 1. 개요

이 문서는 Android 배포를 다음 2개 라인으로 자동화해 운영하기 위한 가이드입니다.

- 내부테스터 자동 배포: `main` 푸시 시 Google Play Internal track 배포
- 릴리스 태그 배포: `vX.Y.Z` 태그 기준 Production draft 업로드

적용 워크플로 파일:

- `.github/workflows/android-internal-distribution.yml`
- `.github/workflows/android-release-from-tag.yml`

## 2. 버전 정책

### 2.1 단일 소스

앱 버전의 기준은 `pubspec.yaml`의 `version` 필드입니다.

- 예: `version: 0.3.0+1`
- `0.3.0` = 사용자 노출 버전명(versionName)
- `+1` = 기본 빌드 번호(로컬 기준)

### 2.2 CI 자동 빌드 번호

CI에서는 다음 규칙으로 빌드 번호를 자동 생성합니다.

- `autoBuildNumber = YYYYMMDD + 2자리 sequence`
- `sequence = (github.run_number + github.run_attempt - 1) % 100`
- 예: `2026030207`

이 방식은 다음을 만족합니다.

- Android `versionCode` 상한(2,100,000,000) 이하 유지
- 기존 날짜 기반 버전코드와 충돌 가능성 완화
- 같은 실행 재시도(run attempt)에서도 sequence가 달라질 수 있도록 반영

### 2.3 버전 올리는 방법

버전명/빌드번호 증가는 스크립트로 처리합니다.

```bash
./scripts/bump_version.sh patch
./scripts/bump_version.sh minor
./scripts/bump_version.sh major
./scripts/bump_version.sh build
```

스크립트는 `pubspec.yaml`의 `version`을 갱신합니다.

- `major|minor|patch`: 버전명 증가 + build number 자동 증가
- `build`: 버전명 유지 + build number 자동 증가
- build number 자동 증가 규칙:
  - `max(현재 build + 1, 현재 epoch second)`
  - Android 상한(2,100,000,000) 초과 시 실패

## 3. CI/CD 워크플로

### 3.1 내부테스터 라인

워크플로: `android-internal-distribution.yml`

- 트리거:
  - `pull_request -> main`: 정적분석/테스트
  - `push -> main`: 분석/테스트 후 내부테스터 배포
  - `workflow_dispatch`: 수동 실행
- 단계:
  1. `flutter analyze`
  2. `flutter test --no-pub`
  3. `flutter build appbundle --release --build-name --build-number`
  4. Google Play `internal` track 업로드

### 3.2 릴리스 태그 라인

워크플로: `android-release-from-tag.yml`

- 트리거:
  - `push tag v*.*.*`
  - `workflow_dispatch`(입력: `tag_version`)
- 검증:
  - 태그 버전(`vX.Y.Z`)과 `pubspec.yaml` 버전명이 동일해야 진행
- 배포:
  - Google Play `production` track, `draft` 상태 업로드

## 4. GitHub Secrets 설정

다음 Secrets를 저장소에 반드시 등록합니다.

| Secret 이름 | 설명 |
|---|---|
| `ANDROID_UPLOAD_KEYSTORE_BASE64` | 업로드용 keystore 파일(base64) |
| `ANDROID_KEYSTORE_PASSWORD` | keystore 비밀번호 |
| `ANDROID_KEY_PASSWORD` | key 비밀번호 |
| `ANDROID_KEY_ALIAS` | 서명 key alias |
| `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` | Play Developer API 서비스계정 JSON |

### 4.1 keystore base64 생성 예시

```bash
base64 -i android/app/upload-keystore.jks | pbcopy
```

복사된 값을 `ANDROID_UPLOAD_KEYSTORE_BASE64`에 저장합니다.

## 5. 운영 절차

### 5.1 내부테스터 배포

1. 기능 작업 후 `main`에 머지/푸시
2. 로컬 내부 테스트 AAB가 필요하면 아래 명령으로 빌드

```bash
./scripts/build_android_internal.sh build
```

3. 버전명까지 함께 올릴 때

```bash
./scripts/build_android_internal.sh patch
```

4. Actions에서 `Android Internal Distribution` 성공 확인
5. Play Console Internal testing 트랙에서 배포 확인

### 5.2 정식 릴리스 준비

1. 버전 업데이트

```bash
./scripts/bump_version.sh minor
```

2. 커밋/푸시
3. 태그 생성/푸시

```bash
git tag vX.Y.Z
git push origin vX.Y.Z
```

4. Actions `Android Release From Tag` 성공 확인
5. Play Console에서 draft 릴리스 검토/게시

## 6. 문제 해결

### 6.1 `Tag/Input version does not match pubspec version`

- 원인: 태그 버전과 `pubspec.yaml` 버전 불일치
- 조치: `pubspec.yaml` 버전과 태그를 동일하게 맞춘 후 재실행

### 6.2 Play 업로드 실패(권한)

- 원인: 서비스계정 권한 부족
- 조치: Play Console에서 앱 권한(릴리스 관리/업로드)을 서비스계정에 부여

### 6.3 빌드 번호 중복

- 현재 정책(`YYYYMMDD + 2자리 sequence`)으로 대부분 해소
- 수동 재배포가 필요한 경우 워크플로 재실행(run attempt 증가) 사용
