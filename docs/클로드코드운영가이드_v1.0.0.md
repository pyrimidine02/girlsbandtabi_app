## 1. 개요

### 1.1 목적

이 문서는 Claude Code를 실제 개발 workflow에 투입할 때 필요한 **운영 원칙**과 **확장 기능 사용 기준**을 정리합니다.

설치 방법이나 화면 설명보다 다음 질문에 답하는 데 집중합니다.

- 어떤 작업은 Claude Code 단일 세션으로 처리하고, 어떤 작업은 역할을 나눠야 하는가
- 어떤 지침은 Agent.md에 두고, 어떤 지침은 Skill로 분리해야 하는가
- 어떤 확장 기능은 전역 설치만 해두고, 어떤 기능만 프로젝트에서 활성화해야 하는가
- Plan Mode, TDD, ADR을 어떤 순서로 묶어야 품질이 올라가는가
- 토큰과 컨텍스트를 낭비하지 않으면서도 재현 가능한 협업 구조를 어떻게 만드는가

### 1.2 적용 범위

이 문서는 다음 항목을 다룹니다.

- Subagent
- Skill
- Plugin
- MCP
- Agent.md
- Teams
- ADR / CHANGELOG / TODO

### 1.3 대상 독자

- 혼자 개발하던 방식에서 Claude Code를 실전 workflow에 붙이려는 개발자
- 반복 작업을 자동화하되 코드 품질과 통제력을 유지하려는 개발자
- 레포마다 다른 성격의 작업을 분리하고 싶은 개발자
- 팀에 공유 가능한 Claude Code 운영 규칙을 만들려는 개발자

### 1.4 문서의 기본 입장

이 문서는 아래 입장을 전제로 합니다.

1. Claude Code는 **바로 코드를 쓰게 하는 도구**가 아니라 **작업 흐름을 설계하고 반복시키는 도구**로 써야 한다.
2. 가장 중요한 것은 기능 소개가 아니라 **운영 규칙**이다.
3. 가장 강하게 권장하는 개발 방식은 **Plan Mode + TDD + 검증 + ADR 기록**이다.
4. 확장 기능은 많이 설치하는 것보다 **좁게 활성화하고 명확하게 분리**하는 쪽이 낫다.
5. Agent.md는 길게 쓰는 문서가 아니라 **항상 적용될 규칙만 남기는 운영 파일**이어야 한다.

## 2. 핵심 운영 원칙

### 2.1 Plan Mode를 기본 진입점으로 둔다

Claude Code를 바로 구현 모드로 던지면 빠르게 진행되지만, 잘못된 방향으로도 빠르게 진행됩니다. 그래서 변경 범위가 조금이라도 있는 작업은 먼저 Plan Mode로 들어가 **읽기 중심 분석**을 한 뒤 구현으로 넘어가는 흐름을 기본값으로 두는 것이 좋습니다.

Plan Mode에서 확인해야 할 것은 다음입니다.

- 어디를 바꿔야 하는가
- 어떤 테스트가 먼저 깨져야 하는가
- 기존 동작과 충돌하는 지점이 있는가
- 데이터 마이그레이션이나 API 계약 영향이 있는가
- 작업을 한 번에 할지, 단계별로 나눌지

Plan Mode는 특히 아래 상황에서 먼저 쓰는 것이 좋습니다.

- 여러 파일을 동시에 수정해야 하는 기능 추가
- 인증/인가, 결제, 캐시, 데이터베이스 스키마처럼 영향 범위가 넓은 변경
- 기존 구조를 이해해야 하는 리팩터링
- 테스트가 부족해서 먼저 안전망을 깔아야 하는 버그 수정

반대로 Plan Mode 없이 바로 진행해도 되는 작업은 매우 작고 국소적인 작업뿐입니다.

- 오타 수정
- 단일 파일 내 문구 교체
- 이미 실패 원인이 명확한 테스트 보정

### 2.2 TDD를 기본 개발 기법으로 둔다

이 문서에서 가장 강하게 권장하는 원칙은 TDD입니다.

Claude Code는 **큰 설계 결정을 인간이 정하고**, 그 아래에서 **작은 반복 루프를 빠르게 수행하는 방식**에 특히 강합니다. 이때 가장 잘 맞는 루프가 바로 TDD입니다.

Claude Code가 TDD에서 강한 이유는 다음과 같습니다.

- 실패하는 테스트 케이스를 빠르게 만들 수 있다.
- 테스트를 통과시키는 최소 구현을 작성하는 반복 작업을 잘 수행한다.
- 테스트 재실행과 회귀 확인을 지치지 않고 반복한다.
- 리팩터링 전후 diff와 테스트 결과를 함께 묶어 설명하기 좋다.
- 개발자가 놓치기 쉬운 경계 조건과 입력 조합을 테스트로 먼저 확장하기 쉽다.

이 문서에서 권장하는 표준 루프는 아래와 같습니다.

1. Plan Mode에서 변경 범위를 정리한다.
2. 실패하는 테스트를 먼저 만든다.
3. 왜 이 테스트가 실패해야 하는지 설명시킨다.
4. 테스트를 통과시키는 최소 구현만 작성한다.
5. 관련 테스트만 먼저 돌린다.
6. 전체 테스트로 회귀를 확인한다.
7. 리팩터링 후 다시 테스트한다.
8. 마지막에 ADR / CHANGELOG / TODO 반영 여부를 정리한다.

즉, Claude Code에게 바로 “기능을 만들어라”라고 시키기보다 아래처럼 요구하는 것이 더 안정적입니다.

- 테스트를 먼저 만들 것
- 최소 구현만 할 것
- 검증 후 리팩터링할 것
- 변경 근거를 남길 것

### 2.3 검증 없는 구현을 완료로 간주하지 않는다

Claude Code를 쓸 때 흔히 생기는 문제는 “변경은 됐지만 검증은 덜 된 상태”를 완료처럼 받아들이는 것입니다. 이 문서는 아래 항목이 끝나기 전까지는 작업이 끝난 것이 아니라고 봅니다.

- 단위 테스트 또는 통합 테스트 통과
- 관련 lint / typecheck 통과
- 회귀 여부 확인
- 변경 근거 정리
- ADR / CHANGELOG / TODO 반영 여부 확인

특히 에이전트가 잘하는 것은 **반복 검증 자동화**입니다. 따라서 검증 단계는 선택이 아니라 설계의 일부로 넣어야 합니다.

### 2.4 전역 설치는 넓게, 프로젝트 활성화는 좁게

많은 확장 기능을 전역으로 설치해 두는 것 자체는 나쁘지 않습니다. 문제는 그것을 **항상 다 켜두는 것**입니다.

Plugin, Agent, Skill, MCP 설명이 동시에 과하게 로드되면 다음 문제가 생깁니다.

- 응답 속도가 느려진다.
- 실제 작업에 쓸 컨텍스트 공간이 줄어든다.
- 현재 프로젝트와 무관한 도구 설명까지 세션에 계속 섞인다.
- Claude가 어떤 도구를 선택해야 하는지 불필요하게 넓은 후보군을 가지게 된다.

따라서 기본 원칙은 아래와 같습니다.

- 전역에는 많이 설치해도 된다.
- 하지만 각 프로젝트에서는 **필요한 것만 활성화**한다.
- 프로젝트별 활성화 여부는 `.claude/settings.local.json` 또는 `.claude/settings.json`에서 제어한다.
- 팀 공용 설정은 프로젝트 설정에, 개인 실험 설정은 local 설정에 둔다.

이 원칙은 생산성보다도 **예측 가능성**을 높입니다. 어떤 레포에서 어떤 플러그인이 실제로 켜져 있는지 명확해야, 동일한 프롬프트가 유사한 결과를 내기 쉽습니다.

### 2.5 문제 정의 없이 확장 기능부터 늘리지 않는다

서브에이전트, 스킬, 플러그인, MCP, 팀즈를 많이 붙인다고 곧바로 생산성이 올라가지는 않습니다.

문제 정의 없이 확장 기능을 늘리면 결과는 보통 아래 둘 중 하나입니다.

- 실제로는 잘 안 쓰는 기능이 계속 컨텍스트만 먹는다.
- 비슷한 역할의 기능이 겹쳐서 운영 기준이 흐려진다.

확장 기능은 아래 질문에 답할 수 있을 때 도입해야 합니다.

- 이 기능이 해결하는 반복 문제가 무엇인가
- Agent.md만으로는 부족한가
- 단일 세션으로는 컨텍스트가 과한가
- 외부 서비스 접근이 정말 필요한가
- 팀에 공유할 가치가 있는가

### 2.6 문서화를 구현 뒤에 붙이는 부수 작업으로 취급하지 않는다

Plan, TDD, 검증, ADR은 따로 노는 요소가 아닙니다. 하나의 작업 흐름으로 연결되어야 합니다.

좋은 작업 흐름은 아래처럼 닫힙니다.

- Plan으로 범위를 정한다.
- TDD로 구현한다.
- 테스트와 검증으로 마무리한다.
- ADR로 왜 그렇게 바꿨는지 남긴다.
- CHANGELOG / TODO로 후속 영향까지 관리한다.

이 흐름이 있어야 Claude Code를 일회성 도우미가 아니라 **반복 가능한 협업 시스템**으로 사용할 수 있습니다.

## 3. 표준 작업 흐름

### 3.1 권장 기본 흐름

가장 권장하는 표준 흐름은 다음과 같습니다.

1. **Plan Mode 진입**
2. **변경 범위와 리스크 식별**
3. **실패 테스트 작성**
4. **최소 구현**
5. **관련 테스트 우선 실행**
6. **전체 회귀 검증**
7. **리팩터링**
8. **ADR / CHANGELOG / TODO 반영**

### 3.2 기능 추가 작업 흐름

기능 추가에서는 아래 순서를 권장합니다.

1. 기존 동작과 변경 포인트 파악
2. API / Domain / DB / UI 중 어느 레이어가 변하는지 분리
3. 실패 테스트 정의
4. 레이어별 최소 구현
5. 테스트 통과
6. 불필요한 중복 제거
7. 결정 사항 문서화

### 3.3 버그 수정 작업 흐름

버그 수정에서는 아래 순서를 권장합니다.

1. 재현 조건을 명확히 적는다.
2. 재현 테스트를 추가한다.
3. 현재 실패를 확인한다.
4. 최소 수정으로 통과시킨다.
5. 인접 기능 회귀 테스트를 돌린다.
6. 근본 원인이 구조적이면 ADR 또는 TODO에 남긴다.

### 3.4 리팩터링 작업 흐름

리팩터링은 기능 추가보다 더 강하게 Plan과 테스트가 필요합니다.

1. 변경하려는 이유를 먼저 정리한다.
2. 동작 보존 테스트를 먼저 확보한다.
3. 파일 단위 / 모듈 단위로 쪼개서 리팩터링한다.
4. 단계마다 테스트를 통과시킨다.
5. 아키텍처 규칙이 바뀌었다면 ADR을 작성한다.

## 4. Subagent 운영 원칙

### 4.1 Subagent의 역할

Subagent는 “작업을 잘게 나눠 다른 역할에게 맡기는 기능”이 아니라, 더 정확히는 **컨텍스트를 분리하고 특정 역할에 국한된 작업을 독립적으로 수행하게 만드는 기능**입니다.

즉, Subagent의 핵심 가치는 병렬성 이전에 **컨텍스트 격리**입니다.

다음과 같은 작업은 Subagent와 잘 맞습니다.

- 코드베이스 탐색
- 특정 모듈만 대상으로 한 리뷰
- 데이터베이스 쿼리 검증
- 보안 관점 점검
- 테스트 관점 점검
- 문서 초안 검토

### 4.2 Subagent를 써야 하는 경우

- 읽어야 할 파일 수가 많아 메인 세션 컨텍스트를 더럽히고 싶지 않을 때
- 특정 역할에만 필요한 도구 제한을 걸고 싶을 때
- 결과보다 과정 격리가 더 중요할 때
- 동일한 검토 패턴을 여러 번 반복할 때
- 구현과 검증을 역할별로 나누고 싶을 때

예를 들면 아래처럼 나누는 것이 좋습니다.

- `code-reviewer`: 구조, 중복, 유지보수성 검토
- `debugger`: 재현 조건, 원인 추적, 수정 후보 비교
- `test-reviewer`: 누락 테스트, 경계 조건, 회귀 위험 검토
- `db-validator`: 인덱스, N+1, migration 영향 검토

### 4.3 Subagent를 쓰지 말아야 하는 경우

- 수정 파일이 한두 개뿐인 단순 작업
- 구현과 검토를 오히려 빠르게 왕복해야 하는 작은 변경
- 역할 분리보다 대화 맥락 유지가 더 중요한 작업
- 순차 의존성이 강해서 병렬성이 거의 없는 작업

Subagent를 쓴다고 항상 더 좋아지는 것은 아닙니다. 작은 작업에서는 오히려 호출 비용과 요약 비용만 늘어날 수 있습니다.

### 4.4 Subagent 설계 원칙

Subagent 정의는 길게 쓰는 것보다 **경계가 분명해야** 합니다.

좋은 Subagent 정의의 특징은 다음과 같습니다.

- 어떤 일을 맡는지 한 문장으로 설명 가능하다.
- 언제 자동 위임되어야 하는지 명확하다.
- 어떤 도구를 써도 되는지 분명하다.
- 무엇을 하면 안 되는지 적혀 있다.
- 출력 형식이 고정되어 있다.

예시:

```markdown
# code-reviewer

목적:

- 변경된 코드의 구조, 중복, 예외 처리, 유지보수성을 검토한다.

주요 점검 항목:

- 중복 로직
- 과도한 함수 길이
- 의미가 불명확한 이름
- 누락된 에러 처리
- 테스트 누락 가능성

출력 형식:

1. 문제 요약
2. 영향 범위
3. 수정 권고
4. 즉시 수정 필요 여부
```

### 4.5 Subagent와 TDD의 결합 방식

Subagent는 TDD와 같이 쓸 때 더 강해집니다.

예를 들어 아래처럼 역할을 나눌 수 있습니다.

- 메인 세션: 작업 계획과 구현 제어
- `test-reviewer`: 먼저 빠진 테스트 케이스 검토
- `code-reviewer`: Green 이후 리팩터링 관점 검토
- `debugger`: 실패 테스트 원인 분석

이 구조는 “하나의 에이전트가 계획, 구현, 검증을 모두 한 번에 맡아서 흐려지는 문제”를 줄여줍니다.

### 4.6 운영 권장안

- Subagent는 **프로젝트에서 반복적으로 쓰는 역할** 위주로 만든다.
- 이름은 역할이 드러나게 짓는다.
- 자동 위임 조건은 좁게 둔다.
- 도구 권한은 필요한 것만 허용한다.
- 구현 전용 / 검증 전용 역할을 분리한다.
- Team까지 갈 필요 없는 범위에서는 Subagent로 해결한다.

메인 본문에서는 아래 정도만 추천 레포로 가져가면 충분합니다.

- `VoltAgent/awesome-claude-code-subagents`
- `rshah515/claude-code-subagents`

상세 목록은 부록에 정리합니다.

## 5. Skill 운영 원칙

### 5.1 Skill의 역할

Skill은 에이전트가 어떤 작업을 반복적으로 처리할 때 필요한 **지식 패키지**이자 **작업 플레이북**입니다.

Agent.md에 넣는 규칙이 “항상 알아야 하는 것”이라면, Skill은 “필요할 때 불러오는 전문 지식”에 가깝습니다.

예를 들면 다음이 Skill에 적합합니다.

- TDD workflow
- systematic debugging
- 문서 생성 규칙
- DB 설계 원칙
- UI 설계 기준
- 보안 점검 checklist
- 특정 외부 서비스 활용법

### 5.2 Skill을 써야 하는 경우

- 반복되는 작업 절차가 있다.
- 사람마다 같은 프롬프트를 반복하고 있다.
- Agent.md에 넣기엔 너무 길지만, 자주 쓰는 지식이다.
- 프로젝트별이 아니라 여러 저장소에서 재사용하고 싶다.
- Slash command 또는 자동 트리거로 호출되면 좋다.

### 5.3 Skill을 Agent.md에 넣지 말아야 하는 경우

아래 내용은 Skill로 빼는 편이 좋습니다.

- 수백 줄짜리 API 레퍼런스
- 상세한 DB 설계 지침
- 특정 framework 전용 작업 절차
- 문서 생성/배포/릴리즈처럼 호출형 workflow
- 테스트/디버깅/리뷰용 단계별 지침

항상 적용될 규칙까지 Skill로 밀어 넣으면, 정작 세션 시작 시 꼭 필요한 기준이 흐려집니다.

### 5.4 TDD Skill을 가장 우선순위 높게 둔다

Skill 중에서 가장 먼저 정리할 만한 것은 TDD 계열입니다.

이 문서의 권장 방향은 명확합니다.

- 새 기능 구현: TDD 우선
- 버그 수정: 재현 테스트 우선
- 리팩터링: 동작 보존 테스트 우선

따라서 Skill 설계도 아래를 강하게 포함해야 합니다.

- 테스트 없이 구현 시작하지 않기
- 실패 테스트를 먼저 만들기
- 최소 구현 이후 바로 검증하기
- 리팩터링 이후 전체 테스트 돌리기
- 테스트 이름과 범위를 명확히 쓰기

Claude Code는 이 흐름을 특히 잘 따릅니다. 사람은 실패 조건과 품질 기준을 정하고, 에이전트는 반복 루프를 정교하게 실행하게 만드는 것이 좋습니다.

### 5.5 Skill 분리 기준

다음 기준으로 분리하면 운영이 쉬워집니다.

- **Reference Skill**: 특정 작업에서 참조할 지식
- **Action Skill**: 직접 호출해 workflow를 수행하는 지침

예시:

- Reference: API 스타일 가이드, React best practices, DB naming rules
- Action: `/tdd-feature`, `/debug-incident`, `/release-check`, `/diagram-system`

### 5.6 Skill 운영 원칙

- Agent.md가 비대해지기 시작하면 Skill로 뺀다.
- 한 Skill은 한 가지 역할만 맡게 한다.
- 프롬프트 템플릿, 체크리스트, 참조 자료를 같이 둔다.
- 스킬 이름만 봐도 언제 쓰는지 알 수 있어야 한다.
- 긴 설명보다 호출 기준과 산출물 형식을 명확히 적는다.

### 5.7 추천 레포

메인 본문에서는 아래 정도만 추천 레포로 언급하면 충분합니다.

- `obra/superpowers`
- `anthropics/skills`
- `sickn33/antigravity-awesome-skills`

기사에서 언급된 세부 레포와 스킬별 정리는 부록에 정리합니다.

## 6. Plugin 운영 원칙

### 6.1 Plugin의 역할

Plugin은 단일 프로젝트에서만 쓰는 설정이 아니라, **재사용 가능한 확장 기능 묶음**을 여러 프로젝트와 팀에 배포할 때 의미가 있습니다.

즉, Plugin은 아래 상황에서 가치가 생깁니다.

- 여러 레포에서 동일한 기능 묶음을 쓰고 싶을 때
- 팀에 공유해야 할 Agent / Skill / Hook / MCP 구성이 있을 때
- 버전 관리와 업데이트 경로가 필요할 때
- 설치와 활성화를 분리하고 싶을 때

### 6.2 Plugin을 써야 하는 경우

- 특정 조직이나 팀의 표준 workflow를 패키징해야 할 때
- 매번 복사하는 `.claude/` 구성이 늘어났을 때
- 개인 스크립트를 넘어서 팀 자산이 되었을 때
- 여러 기능을 이름공간과 함께 배포해야 할 때

### 6.3 Plugin 대신 로컬 설정으로 끝내도 되는 경우

- 아직 실험 중인 workflow
- 특정 프로젝트에만 필요한 규칙
- 레포 성격상 다른 프로젝트에 재사용되지 않는 지식
- 빠르게 수정하고 버리는 규칙

처음부터 모든 것을 Plugin으로 만들 필요는 없습니다. 보통은 **로컬에서 검증 → 프로젝트 공용 → Plugin 패키징** 순서가 더 안정적입니다.

### 6.4 Plugin 과다 활성화 문제

Plugin은 많이 설치해둘 수 있지만, 프로젝트마다 실제로 켤 항목은 좁혀야 합니다.

특히 `settings.json`에서 활성화된 플러그인이 많으면, 각 플러그인의 설명과 도구 정의가 세션 컨텍스트에 섞이면서 다음 문제가 생길 수 있습니다.

- 현재 작업과 무관한 확장 기능이 지속적으로 로드됨
- 도구 선택 비용 증가
- 응답 속도 저하
- 실제 작업에 사용할 컨텍스트 공간 감소

따라서 아래 원칙을 권장합니다.

- 전역 설치는 넓게
- 프로젝트 활성화는 좁게
- 개인 실험은 `.claude/settings.local.json`
- 팀 공용 활성화는 `.claude/settings.json`

### 6.5 프로젝트별 override 권장

다음 예시는 많이 설치해 둔 Plugin 중 현재 Flutter 프로젝트에서 필요한 것만 남기는 방식입니다.

```json
{
  "enabledPlugins": {
    "flutter-tools@team-tools": true,
    "mobile-testing@team-tools": true,
    "deployment@team-tools": false,
    "infra-kubernetes@team-tools": false,
    "data-pipeline@personal": false,
    "security-audit@personal": false
  }
}
```

이 방식의 장점은 명확합니다.

- 내 머신에는 많은 plugin이 있어도 된다.
- 하지만 현재 레포에서는 관련 없는 기능을 꺼서 context를 아낄 수 있다.
- 실험 중인 plugin은 local scope에서만 다뤄 팀 설정을 오염시키지 않는다.

### 6.6 Plugin 설계 원칙

- 하나의 Plugin은 하나의 문제 영역을 맡게 한다.
- 이름만 보고 역할이 드러나야 한다.
- Agent / Skill / Hook / MCP를 무리하게 한 Plugin에 몰아넣지 않는다.
- 여러 Plugin을 조합할 수 있게 작게 나눈다.
- 항상 로드돼야 하는 규칙은 Plugin이 아니라 Agent.md 쪽에 둔다.

### 6.7 운영 권장안

- 팀 단위로 반복되는 workflow만 Plugin으로 승격한다.
- 실험 중인 기능은 Plugin이 아니라 local 설정에서 검증한다.
- 각 레포에서는 필요한 Plugin만 켠다.
- “깔아두기”와 “활성화하기”를 구분한다.

### 6.8 추천 자료

Plugin 섹션은 공식 문서와 아래 레포 정도를 기준으로 보면 충분합니다.

- 공식: Claude Code Plugins 문서
- 추천 레포: `anthropics/claude-plugins-official`, `wshobson/agents`

상세 내용은 부록에 정리합니다.

## 7. MCP 운영 원칙

### 7.1 MCP의 역할

MCP는 Claude Code가 외부 도구, 데이터, API, 서비스에 접근하도록 연결하는 계층입니다.

즉, MCP는 “더 똑똑한 프롬프트”가 아니라 **실제 외부 시스템 접근 권한**을 여는 기능입니다.

예를 들면 다음과 같은 경우가 MCP와 맞습니다.

- GitHub 이슈/PR 조회 및 조작
- 데이터베이스 질의
- 모니터링 툴 조회
- 디자인 툴 연동
- 슬랙/노션/지라/사내 API 접근

### 7.2 MCP를 써야 하는 경우

- 코드베이스 바깥의 데이터가 실제로 필요할 때
- 문서나 티켓 시스템과 구현 작업을 직접 연결해야 할 때
- 사람 손으로 옮기던 운영 정보를 에이전트가 직접 읽어야 할 때
- 외부 시스템 기반 검증을 workflow에 넣고 싶을 때

### 7.3 MCP를 함부로 붙이지 말아야 하는 경우

- 그냥 “있으면 좋아 보이는” 도구 연결
- 실사용 시나리오 없이 가능한 것만 늘리는 연결
- 민감한 시스템에 대한 접근 범위 정의 없이 추가하는 연결
- 하나의 질문에 필요하지 않은 서버를 여러 개 동시에 활성화하는 구성

MCP는 강력하지만, 동시에 위험한 계층이기도 합니다. 외부 도구 접근은 곧 권한 문제와 prompt injection 문제로 이어질 수 있습니다.

### 7.4 MCP 운영 원칙

- 꼭 필요한 서버만 연결한다.
- 읽기 전용과 변경 가능 서버를 구분한다.
- 민감한 시스템은 allowlist / denylist 기준을 둔다.
- 도구 검색 범위를 좁혀 선택 비용을 줄인다.
- 서버를 많이 두기보다 실제로 쓰는 것만 유지한다.

### 7.5 MCP와 Plugin의 관계

MCP는 도구 연결이고, Plugin은 그 도구 연결을 포함한 배포 단위가 될 수 있습니다.

즉, 아래처럼 생각하면 됩니다.

- 외부 시스템 접근 자체: MCP
- 그 접근을 Agent / Skill / Hook과 함께 재사용 가능한 묶음으로 배포: Plugin

### 7.6 운영 권장안

- 먼저 프로젝트에서 필요한 외부 시스템이 무엇인지 좁힌다.
- 그 다음 읽기 전용 시나리오부터 붙인다.
- 쓰기 권한은 최소화한다.
- MCP 도입 전후로 어떤 workflow가 줄어드는지 명확히 한다.

### 7.7 추천 자료

MCP 섹션은 공식 문서와 아래 레포 정도를 기준으로 보면 충분합니다.

- 공식: Claude Code MCP 문서
- 추천 레포: `github/github-mcp-server`, `zilliztech/claude-context`

상세 내용은 부록에 정리합니다.

## 8. Agent.md 운영 원칙

### 8.1 Agent.md의 역할

Agent.md는 프로젝트에서 Claude Code가 **항상 알고 있어야 하는 규칙**을 담는 문서입니다.

이 문서에 들어가야 하는 것은 “긴 설명”이 아니라 다음과 같은 항목입니다.

- 빌드 / 테스트 / lint / typecheck 명령
- 아키텍처 경계
- 네이밍 규칙
- 금지 사항
- 보안 규칙
- 리뷰 기준
- 문서화 규칙
- TDD 원칙
- ADR 작성 기준

### 8.2 Agent.md에 넣어야 하는 것

아래 항목은 Agent.md에 두는 것이 좋습니다.

- 항상 적용될 규칙
- 팀 공용 기준
- 작업마다 반복 설명하기 싫은 기본 원칙
- 코드 스타일보다 상위의 workflow 규칙
- “무조건 지켜야 하는 것”과 “하지 말아야 하는 것”

예시:

- 테스트 없이 구현 시작하지 말 것
- 큰 변경은 Plan을 먼저 제시할 것
- 인증/결제/스키마 변경 시 ADR 작성할 것
- PR 전 lint / test / typecheck를 모두 통과시킬 것
- TODO를 남기면 제거 기준을 같이 적을 것

### 8.3 Agent.md에 넣지 말아야 하는 것

아래 내용은 Agent.md를 빠르게 비대하게 만듭니다.

- 상세 API 문서
- 프레임워크별 세세한 레퍼런스
- 잘 사용하지 않는 긴 예시
- 한 번성 실험 메모
- 특정 작업에서만 쓰는 긴 playbook

이런 내용은 Skill이나 별도 문서로 빼는 것이 낫습니다.

### 8.4 좋은 Agent.md의 형태

첨부한 AGENTS 문서 스타일에서 가져갈 핵심은 다음입니다.

- 목적이 먼저 나온다.
- 원칙이 짧고 단단하다.
- 역할 분리와 승인 규칙이 있다.
- 품질 기준과 체크리스트가 있다.
- 문서화 의무가 workflow의 일부로 포함된다.

즉, Agent.md는 설명문이 아니라 **작업 기준서**처럼 읽혀야 합니다.

### 8.5 Agent.md에 반드시 넣을 항목

이 문서의 권장 기본 항목은 아래와 같습니다.

1. Goals
2. Approval Rules
3. Plan Rules
4. TDD Rules
5. Review Checklist
6. Security Rules
7. Architecture Boundaries
8. Documentation Rules
9. ADR / CHANGELOG / TODO Rules

### 8.6 TDD와 ADR을 Agent.md 수준에서 강제한다

Agent.md 차원에서 아래 규칙을 두는 것을 권장합니다.

- 새 기능은 실패 테스트부터 시작한다.
- 버그 수정은 재현 테스트부터 만든다.
- 큰 변경은 구현 전 Plan을 남긴다.
- 구조적 결정은 ADR로 남긴다.
- 작업 완료 전 CHANGELOG / TODO 반영 여부를 확인한다.

즉, Agent.md는 단지 코드 스타일 문서가 아니라 **품질 시스템의 엔트리 포인트**여야 합니다.

### 8.7 권장 템플릿

```markdown
## Goals

- 테스트 우선
- 회귀 없는 변경
- 구조적 결정의 기록

## Plan Rules

- 다중 파일 변경, 스키마 변경, 인증/인가 변경은 Plan 먼저

## TDD Rules

- 구현 전에 실패 테스트 작성
- 최소 구현 후 관련 테스트 실행
- 리팩터링 후 전체 회귀 확인

## ADR Rules

- 모듈 경계 변경
- 데이터 모델 변경
- 인프라/보안 정책 변경
- 외부 연동 방식 변경
  은 ADR 작성

## Review Checklist

- 테스트 통과
- lint / typecheck 통과
- 중복 로직 제거
- 예외 처리 확인
- CHANGELOG / TODO / ADR 반영 여부 확인
```

## 9. Teams 운영 원칙

### 9.1 Teams의 역할

Teams는 여러 Claude Code 세션이 서로 독립적으로 일하면서 작업을 나누고 조정하는 기능입니다.

Subagent와의 가장 큰 차이는 다음입니다.

- Subagent: 메인 세션 내부에서 호출되는 집중 작업자
- Teams: 서로 독립적인 세션이 협업하는 구조

즉, Teams는 단순한 역할 분리를 넘어서 **병렬 조사, 상호 검토, 분업 조정**이 필요한 경우에 의미가 있습니다.

### 9.2 Teams를 써야 하는 경우

- 병렬 탐색의 이점이 큰 작업
- 서로 다른 가설을 동시에 검증해야 하는 디버깅
- 프론트 / 백엔드 / 테스트가 동시에 움직이는 기능 개발
- 리뷰 관점을 병렬로 분리하고 싶은 경우
- 구현보다 조사와 비교가 더 중요한 작업

예시:

- `research` 팀: 관련 코드, 외부 문서, 기존 이슈를 병렬 조사
- `debug` 팀: 원인 가설 3개를 동시에 검증
- `feature` 팀: API, UI, 테스트를 병렬 분리
- `security` 팀: 인증, 의존성, 비밀정보, 권한 흐름을 병렬 점검

### 9.3 Teams를 쓰지 말아야 하는 경우

- 순차 의존성이 강한 작업
- 같은 파일을 계속 수정해야 하는 작업
- 작은 버그 수정
- 병렬성보다 맥락 일관성이 더 중요한 작업
- 팀 조정 비용이 실제 이득보다 큰 작업

Teams는 강력하지만 토큰 사용량도 크고 조정 비용도 있습니다. 그래서 **병렬 탐색 가치가 큰 경우**에만 쓰는 것이 좋습니다.

### 9.4 Teams 운영 원칙

- Team lead가 역할과 산출물을 명확히 나눈다.
- 동일 파일 동시 편집을 피한다.
- 팀원별 책임 범위를 좁힌다.
- 결과 합성 전에 각 팀원의 검증 기준을 맞춘다.
- Plan 승인 기준을 팀 차원에서 둔다.

### 9.5 Teams와 TDD의 결합 방식

Teams는 구현 병렬화보다도 **검증 병렬화**에 더 잘 맞는 경우가 많습니다.

예를 들면 아래 구조가 좋습니다.

- 팀장: Plan 정리, 범위 결정, 최종 합성
- 구현 담당: 최소 구현
- 테스트 담당: 실패 테스트 / 회귀 테스트 설계
- 리뷰 담당: 구조, 성능, 보안 검토

이 구조는 사람이 모든 검토를 직접 반복하지 않아도 되게 해 줍니다.

### 9.6 Teams와 ADR

팀 단위로 조사와 구현이 병렬로 일어난 작업은, 나중에 “왜 이 방향을 택했는지”를 잃어버리기 쉽습니다. 따라서 Teams를 쓴 작업은 ADR 작성 우선순위를 더 높게 보는 것이 좋습니다.

- 어떤 대안을 검토했는가
- 왜 현재 안을 선택했는가
- 남은 제약이 무엇인가
- 후속 작업이 필요한가

이 네 가지는 최소한 ADR에 남겨야 합니다.

### 9.7 추천 자료

Teams 섹션은 공식 agent teams 문서와 아래 레포 정도를 보면 충분합니다.

- 공식: Claude Code Agent Teams 문서
- 추천 레포: `wshobson/agents`, `cs50victor/claude-code-teams-mcp`

상세 내용은 부록에 정리합니다.

## 10. ADR / CHANGELOG / TODO 운영 규칙

### 10.1 ADR을 언제 작성해야 하는가

다음 변경은 ADR 대상으로 보는 것이 좋습니다.

- 모듈 경계 변경
- 인증/인가 정책 변경
- 데이터 모델 및 스키마 구조 변경
- 주요 성능 최적화 전략 변경
- 외부 시스템 연동 방식 변경
- 배포 / 운영 / 보안 정책 변경
- Team 구조나 Subagent 역할 체계처럼 개발 workflow 자체를 바꾸는 경우

### 10.2 ADR의 목적

ADR은 “형식적으로 남기는 기록”이 아니라 아래를 위해 씁니다.

- 왜 이 구조를 택했는지 잊지 않기 위해
- 다른 대안을 검토했는지 남기기 위해
- 이후 리팩터링 시 맥락을 재구성하기 위해
- Claude Code와 인간 모두가 같은 결정을 다시 참조하기 위해

### 10.3 ADR 최소 구성

다음 네 섹션은 반드시 있는 편이 좋습니다.

- Status
- Context
- Decision
- Consequences

### 10.4 작업 흐름 안에서 ADR을 넣는 위치

ADR은 작업이 끝난 뒤 별도 문서 작업이 아니라, 아래 흐름의 마지막 단계로 넣는 것이 좋습니다.

1. Plan
2. TDD
3. 구현
4. 검증
5. ADR
6. CHANGELOG / TODO

### 10.5 CHANGELOG와 TODO

작업 후 아래 기준을 지키면 운영이 안정됩니다.

- **CHANGELOG**: 실제 바뀐 파일과 영향 범위를 남긴다.
- **TODO**: 남겨진 기술 부채와 제거 기준을 남긴다.
- **ADR**: 구조적 결정과 이유를 남긴다.

즉, 세 문서는 역할이 다릅니다.

- CHANGELOG = 무엇이 바뀌었는가
- TODO = 무엇이 아직 남았는가
- ADR = 왜 그렇게 바꾸었는가

### 10.6 권장 규칙

- 작은 코드 수정까지 모두 ADR로 만들 필요는 없다.
- 하지만 구조 변경인데 ADR이 없으면 나중에 재해석 비용이 커진다.
- TODO에는 제거 기준을 같이 적는다.
- CHANGELOG는 파일명 중심으로 남긴다.
- Claude Code 작업 완료 프롬프트에 ADR / TODO / CHANGELOG 반영 여부를 항상 포함한다.

### 10.7 ADR 예시 템플릿

```markdown
# ADR-YYYYMMDD: 인증 플로우를 이메일 인증 기반으로 변경

**Status:** Accepted
**Date:** YYYY-MM-DD
**Deciders:** 개발팀 손호영
**Tags:** auth, user, onboarding

## Context

기존 가입 플로우는 이메일 소유 확인 없이 계정을 활성화한다.
운영 중 스팸 계정과 잘못된 이메일 등록 문제가 반복되었다.

## Decision

가입 직후 inactive 상태를 부여하고, 이메일 인증 완료 후 활성화한다.
로그인 단계에서 미인증 계정을 차단한다.

## Consequences

- Positive: 잘못된 계정 생성 감소
- Neutral: 가입 플로우 복잡도 증가
- Negative: 기존 테스트 및 세션 처리 로직 수정 필요
```

## 11. 실전 프롬프트 예시

### 11.1 Plan Mode로 시작

```text
Plan mode로 진행해.
이 작업에서 변경될 파일, 테스트 전략, 리스크, 롤백 포인트를 먼저 정리해.
바로 구현하지 말고 계획부터 검토할 수 있게 작성해.
```

### 11.2 기능 추가를 TDD로 진행

```text
이 기능은 TDD로 진행해.
1) 실패하는 테스트부터 만들고
2) 왜 실패하는지 설명하고
3) 최소 구현으로 통과시키고
4) 리팩터링 후
5) 관련 테스트와 전체 회귀 결과를 정리해.
```

### 11.3 버그 수정을 TDD로 진행

```text
바로 수정하지 말고 재현 테스트부터 추가해.
현재 실패를 보여준 다음 최소 수정으로 통과시키고,
관련 회귀 테스트까지 확인해.
```

### 11.4 Subagent로 코드 리뷰 분리

```text
구현 후 code-reviewer 서브에이전트를 사용해
중복 로직, 과도한 함수 길이, 누락된 에러 처리,
테스트 누락 가능성을 검토해.
수정이 필요한 항목은 영향 범위까지 같이 정리해.
```

### 11.5 Subagent로 디버깅 분리

```text
debugger 서브에이전트를 사용해
이 실패의 재현 조건, 가능한 원인 가설, 로그/테스트로 검증할 지점을 정리해.
추측만 하지 말고 가장 가능성 높은 원인부터 우선순위를 매겨줘.
```

### 11.6 Teams로 병렬 탐색

```text
이 작업은 team으로 나눠 진행해.
- 한 명은 인증 흐름 조사
- 한 명은 DB 영향 분석
- 한 명은 테스트 전략 정리
각자 결과를 요약한 뒤, 최종 구현 순서를 하나의 계획으로 합쳐줘.
```

### 11.7 Plugin 활성화 최소화

```text
현재 프로젝트에서 실제로 필요한 plugin만 남기고 싶다.
설치된 plugin 목록 기준으로 이 레포에 불필요한 항목을 분류하고,
.settings.local 기준 override 예시를 제안해줘.
```

### 11.8 ADR 작성 요청

```text
이번 변경은 구조적 결정이 있으니 ADR 초안을 작성해.
Context / Decision / Consequences 형식으로 정리하고,
대안과 트레이드오프도 한 문단씩 포함해.
```

### 11.9 작업 마무리 확인

```text
작업 마무리 체크를 해줘.
- 테스트 통과 여부
- lint / typecheck 여부
- ADR 필요 여부
- CHANGELOG 반영 포인트
- TODO로 남겨야 할 항목
을 정리해.
```

## 12. 권장 도입 순서

### 12.1 1단계: 기본 운영 규칙 확립

가장 먼저 할 일은 확장 기능 추가가 아니라 아래를 정하는 것입니다.

- Plan Mode를 기본 진입점으로 둘 것
- TDD를 기본 구현 방식으로 둘 것
- Agent.md에 항상 적용될 규칙을 넣을 것
- ADR / CHANGELOG / TODO 반영 규칙을 정할 것

### 12.2 2단계: Skill 정리

반복 프롬프트가 보이기 시작하면 Skill로 분리합니다.

가장 먼저 추천하는 것은 다음입니다.

- TDD
- systematic debugging
- code review
- architecture / API design
- documentation

### 12.3 3단계: Subagent 분리

다음 역할이 반복되면 Subagent를 둡니다.

- code review
- debugging
- testing
- database validation
- docs review

### 12.4 4단계: Plugin 승격

프로젝트를 넘어 여러 레포에서 재사용할 구성이 되면 Plugin으로 승격합니다.

### 12.5 5단계: MCP 연결

실제로 외부 시스템이 필요한 workflow가 있을 때만 붙입니다.

### 12.6 6단계: Teams 도입

마지막 단계에서만 Teams를 도입합니다. Teams는 가장 늦게 도입해도 됩니다. Plan, TDD, Skill, Subagent만 잘 운영해도 대부분의 레포에서는 충분합니다.
