## 1) 요청 배경

### 1.1 알림 설정 저장 불가 이슈(현재 장애)
모바일 앱에서 `PUT /api/v1/notifications/settings` 요청이 지속적으로 `409 CONFLICT`를 반환하여,
사용자가 알림 토글을 변경해도 저장이 완료되지 않는 상태입니다.

재현 로그(발췌):
- 1차 PUT 409: `requestId=req_1773080021445_2560`
- 즉시 GET 200 후 동일 사용자 의도값으로 2차 PUT 재시도
- 2차 PUT도 409: `requestId=req_1773080021744_9136`

즉, **"새로고침 후 재시도"를 수행해도 저장 성공으로 수렴하지 않는 상태**입니다.

### 1.2 커뮤니티 소셜 알림 요구
사용자 요구사항:
1. 팔로우한 사용자가 새 글 작성 시 알림
2. 내가 작성한 글에 댓글이 달리면 알림
3. 내가 작성한 댓글에 답글이 달리면 알림

## 2) 백엔드 요청사항 A — 알림 설정 409 충돌 해소

## 2.1 권장 계약 (Optimistic Lock 명시)
현재 응답에는 동시성 토큰이 없어 클라이언트가 올바른 재시도를 구성하기 어렵습니다.
아래 계약으로 명시 부탁드립니다.

### GET /api/v1/notifications/settings 응답 확장
```json
{
  "pushEnabled": true,
  "emailEnabled": false,
  "categories": ["LIVE_EVENT", "FAVORITE", "COMMENT"],
  "version": 12,
  "updatedAt": "2026-03-10T03:13:41.641Z"
}
```

### PUT /api/v1/notifications/settings 요청
```json
{
  "pushEnabled": true,
  "emailEnabled": false,
  "categories": ["FAVORITE"],
  "version": 12
}
```

### 성공 응답
```json
{
  "pushEnabled": true,
  "emailEnabled": false,
  "categories": ["FAVORITE"],
  "version": 13,
  "updatedAt": "2026-03-10T03:13:42.001Z"
}
```

### 충돌(409) 응답 상세
```json
{
  "error": {
    "code": "NOTIFICATION_SETTINGS_VERSION_CONFLICT",
    "message": "다른 요청에서 리소스를 갱신했습니다.",
    "details": {
      "expectedVersion": 12,
      "currentVersion": 13,
      "current": {
        "pushEnabled": true,
        "emailEnabled": false,
        "categories": ["LIVE_EVENT", "FAVORITE"],
        "version": 13,
        "updatedAt": "2026-03-10T03:13:41.641Z"
      }
    }
  }
}
```

## 2.2 최소 요구사항(단기)
아래 중 하나는 반드시 필요합니다.
- A안: 위 `version` 계약 도입
- B안: 동일 사용자 단일 리소스(`notifications/settings`)는 last-write-wins 처리로 409 미반환

> 현재는 새로고침+재시도 이후에도 409가 반복되어 실사용 저장이 불가능합니다.

## 2.3 오류 코드 표준화 요청
- `NOTIFICATION_SETTINGS_VERSION_CONFLICT`
- `NOTIFICATION_SETTINGS_INVALID_CATEGORY`
- `NOTIFICATION_SETTINGS_UPDATE_FAILED`

## 3) 백엔드 요청사항 B — 커뮤니티 소셜 알림 발행 확장

## 3.1 신규/명확화 이벤트
| eventCode | 설명 | 권장 카테고리 | deeplink |
|---|---|---|---|
| `FOLLOWING_POST_CREATED` | 팔로우한 사용자의 새 게시글 | `FOLLOWING_POST` (신규) | `/board/posts/{postId}` |
| `MY_POST_COMMENT_CREATED` | 내 게시글에 새 댓글 | `COMMENT` | `/board/posts/{postId}` |
| `MY_COMMENT_REPLY_CREATED` | 내 댓글에 새 답글 | `COMMENT` | `/board/posts/{postId}` |

## 3.2 수신자 규칙
- `FOLLOWING_POST_CREATED`
  - 수신자: 작성자 팔로워
  - 제외: 작성자 본인, 차단 관계 사용자
- `MY_POST_COMMENT_CREATED`
  - 수신자: 게시글 작성자
  - 제외: 댓글 작성자가 본인인 경우
- `MY_COMMENT_REPLY_CREATED`
  - 수신자: 부모 댓글 작성자
  - 제외: 답글 작성자가 본인인 경우

## 3.3 중복/소음 제어
- 동일 `recipientId + eventCode + entityId` 기준 dedupe window 적용 (권장 10~30초)
- 대량 발생 시 요약(digest) 정책 옵션 제공

## 3.4 알림 채널 정합성
- 발행 대상: Push + SSE + `/api/v1/notifications` 목록 저장
- `notificationId` 단일 식별자로 채널 간 정합성 보장

## 4) 알림 설정 카테고리 확장 요청

현재 카테고리:
- `LIVE_EVENT`, `FAVORITE`, `COMMENT`

요청:
- `FOLLOWING_POST` 추가

예시:
```json
{
  "pushEnabled": true,
  "emailEnabled": false,
  "categories": ["LIVE_EVENT", "FAVORITE", "COMMENT", "FOLLOWING_POST"]
}
```

> FE는 `FOLLOWING_POST` 토글 UI를 후속 배포에서 노출할 예정이며,
> 그 전까지는 서버 기본 정책(ON/OFF) 가이드가 필요합니다.

## 5) payload 계약(권장)

### Push/SSE payload 공통 키
- `notificationId` (필수)
- `notificationType` 또는 `type`
- `title`
- `body`
- `deeplink`
- `targetId` (postId/commentId 등)
- `projectCode` (선택)

예시:
```json
{
  "notificationId": "ntf_20260310_0001",
  "notificationType": "FOLLOWING_POST_CREATED",
  "title": "팔로우한 사용자가 새 글을 올렸어요",
  "body": "Content Manager: 추천 피드",
  "deeplink": "/board/posts/41680e8e-d4ca-43b8-b5bc-cc848cf9e166",
  "targetId": "41680e8e-d4ca-43b8-b5bc-cc848cf9e166",
  "projectCode": "bang-dream"
}
```
