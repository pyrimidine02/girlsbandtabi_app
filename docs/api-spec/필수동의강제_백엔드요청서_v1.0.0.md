# 필수동의강제_백엔드요청서_v1.0.0

작성일: 2026-03-08  
대상: GirlsBandTabi Backend (Identity/User Privacy)

---

## 1) 목적
- 로그인 사용자 중 `이용약관` / `개인정보 처리방침` 미동의자는 서비스 이용을 차단하는 정책을
  서버 기준으로 일관되게 강제하기 위함.
- 현재 앱은 전역 차단 팝업(클라이언트)을 적용했으나, 동의 저장이 로컬 fallback 중심이라
  멀티디바이스/감사 이력 정합성이 부족함.

---

## 2) 현재 상태(클라이언트)
- 앱은 로그인 후 전역으로 필수 동의 상태를 검사함.
- 동의 조회: `GET /api/v1/users/me/consents` (기존)
- 미동의 시: 전역 차단 팝업 노출, 동의 전 서비스 사용 불가
- 문제:
  - 필수 동의 제출용 서버 API 부재(현재 조회 중심)
  - 서버 감사 이력과 앱 상태가 디바이스별로 어긋날 수 있음

---

## 3) 백엔드 요청사항 (핵심)

### 3.1 필수 동의 제출 API 추가 (필수)
- `POST /api/v1/users/me/consents`
- Auth: 필요
- 용도: 팝업에서 체크한 필수 동의를 서버 이력에 즉시 반영

요청 예시:
```json
{
  "consents": [
    {
      "type": "TERMS_OF_SERVICE",
      "version": "v2026.03.06",
      "agreed": true,
      "agreedAt": "2026-03-08T14:10:00Z"
    },
    {
      "type": "PRIVACY_POLICY",
      "version": "v2026.03.06",
      "agreed": true,
      "agreedAt": "2026-03-08T14:10:00Z"
    }
  ]
}
```

응답 예시:
```json
{
  "success": true,
  "statusCode": 200,
  "data": {
    "saved": [
      {"type": "TERMS_OF_SERVICE", "version": "v2026.03.06", "agreedAt": "2026-03-08T14:10:00Z"},
      {"type": "PRIVACY_POLICY", "version": "v2026.03.06", "agreedAt": "2026-03-08T14:10:00Z"}
    ]
  }
}
```

요청:
- 멱등성 보장(동일 요청 재전송 시 중복 insert 방지)
- `agreed=false`는 필수동의 API에서는 거부(400) 또는 별도 정책 정의

---

### 3.2 필수 동의 상태 API 추가 (권장)
- `GET /api/v1/users/me/consent-status`
- Auth: 필요
- 용도: 앱 시작/로그인 직후, 필수 동의 차단 여부를 단일 API로 즉시 판정

응답 예시:
```json
{
  "success": true,
  "statusCode": 200,
  "data": {
    "canUseService": false,
    "requiredConsents": [
      {
        "type": "TERMS_OF_SERVICE",
        "requiredVersion": "v2026.03.06",
        "agreed": true,
        "agreedVersion": "v2026.03.06",
        "agreedAt": "2026-03-08T14:10:00Z",
        "needsReconsent": false
      },
      {
        "type": "PRIVACY_POLICY",
        "requiredVersion": "v2026.03.06",
        "agreed": false,
        "agreedVersion": null,
        "agreedAt": null,
        "needsReconsent": true
      }
    ]
  }
}
```

요청:
- `canUseService=false`이면 클라이언트 차단 팝업 유지
- 버전 변경 시 `needsReconsent=true`로 명확히 내려주기

---

### 3.3 정책 메타데이터 API 정합성
- 기존/예정 `policies metadata` API와 필수 동의 타입/버전 정합 필요
- 최소 필요 필드:
  - `type`
  - `version`
  - `effectiveAt`
  - `requiresReconsent`

---

## 4) 에러 코드 요청
- `CONSENT_INVALID_TYPE`
- `CONSENT_VERSION_INVALID`
- `CONSENT_REQUIRED_FIELDS_MISSING`
- `CONSENT_SUBMISSION_CONFLICT` (중복/멱등 충돌 시)
- `CONSENT_STATUS_UNAVAILABLE`

클라이언트는 `error.code`/`error.message`를 그대로 로깅 및 사용자 메시지 분기함.

---

## 5) 하위호환/마이그레이션
- 기존 `GET /api/v1/users/me/consents`는 유지(이력 화면용).
- 신규 `GET /consent-status`가 준비되면 앱은 로그인 직후 상태 판정을 해당 API로 전환.
- 신규 `POST /consents` 반영 전까지는 앱이 로컬 fallback으로 임시 동작 중.

---

## 6) QA 체크리스트
1. 필수 미동의 계정: `consent-status.canUseService=false` 확인
2. 팝업 동의 후 `POST /consents` 성공 및 즉시 `canUseService=true` 전환 확인
3. 동일 payload 재전송 시 중복 이력 폭증 없이 멱등 처리 확인
4. 정책 버전 상향 시 기존 동의 계정이 `needsReconsent=true`로 변환되는지 확인
5. `GET /consents` 이력과 `GET /consent-status` 최신 상태가 일치하는지 확인

---

## 7) 구현 우선순위 제안
1. `POST /api/v1/users/me/consents`  
2. `GET /api/v1/users/me/consent-status`  
3. 정책 메타데이터 응답 정합화 (`requiresReconsent`)
