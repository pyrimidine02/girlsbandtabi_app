## 1) 목적
- 현재 앱 기준으로 실제 사용자 체감이 좋은 알림 주제를 우선 발행하고,
  탭 동작(딥링크)과 알림함 동기화를 안정적으로 맞추기 위함.
- 푸시 발행 기준/문구/페이로드를 서버와 명확히 계약해 중복, 오발송,
  라우팅 실패를 줄이기 위함.

---

## 2) 현재 앱 지원 범위 (서버가 맞춰야 할 부분)

### 2.1 사용자 알림 설정 카테고리
- `LIVE_EVENT`
- `FAVORITE`
- `COMMENT`

> 서버 요청: 카테고리별 opt-in/opt-out 필터링 후 enqueue 해주세요.

### 2.2 클라이언트가 읽는 payload 키
- 식별: `notificationId` (필수)
- 타입: `type` 또는 `notificationType`
- 제목: `title` 또는 `notificationTitle` 또는 `subject`
- 본문: `body` 또는 `message` 또는 `content`
- 딥링크: `deeplink` 또는 `deepLink`
- 대상 ID: `targetId` 또는 `entityId` 또는 `contentId`
- 프로젝트: `projectCode` 또는 `projectId`
- 선택: `actionUrl`

### 2.3 딥링크 라우팅 지원 경로
- `/board/posts/{postId}`
- `/live/{eventId}`
- `/places/{placeId}`
- `/info/news/{newsId}`
- `/users/{userId}`
- `/notifications`

> 서버 요청: 위 경로 중 하나를 `deeplink`로 내려주세요.

## 3) 서버 발행 정책 요청 (핵심)
1. **멱등성 보장**
   - 동일 알림은 동일 `notificationId` 사용
   - 중복 enqueue/중복 저장 방지
2. **알림함/푸시 정합성**
   - 푸시 payload와 `/api/v1/notifications` 목록의 `id`를 동일 `notificationId`로 맞춤
3. **카테고리 필터링**
   - 사용자 설정(`LIVE_EVENT`, `FAVORITE`, `COMMENT`) 미허용 시 푸시/알림함 모두 미발행
4. **우선순위**
   - 시간 민감(`LIVE_EVENT_*`)은 high priority
   - 일반 업데이트는 normal priority
5. **빈번 알림 제한**
   - 동일 사용자·동일 엔티티 기준 rate limit 적용(예: 댓글 폭주 시 digest 또는 배치)

## 4) 추천 발행 시나리오/문구 템플릿

| eventCode | category | 트리거 | 제목 예시 | 본문 예시 | deeplink | entityId |
|---|---|---|---|---|---|---|
| `LIVE_EVENT_REMINDER_D1` | `LIVE_EVENT` | 공연 D-1 | 내일 라이브가 있어요 | 3/9 19:00 Zepp Osaka, 일정 확인해보세요 | `/live/{eventId}` | `{eventId}` |
| `LIVE_EVENT_REMINDER_H1` | `LIVE_EVENT` | 공연 1시간 전 | 라이브 시작 1시간 전 | 입장 전 위치/동선을 미리 확인하세요 | `/live/{eventId}` | `{eventId}` |
| `LIVE_EVENT_UPDATED` | `LIVE_EVENT` | 시간/장소 변경 | 라이브 일정이 변경됐어요 | 변경된 시간과 장소를 확인해주세요 | `/live/{eventId}` | `{eventId}` |
| `LIVE_EVENT_CANCELED` | `LIVE_EVENT` | 취소 공지 | 라이브 일정이 취소됐어요 | 취소 공지와 대체 일정을 확인하세요 | `/live/{eventId}` | `{eventId}` |
| `FAVORITE_PLACE_UPDATE` | `FAVORITE` | 즐겨찾기 장소 신규 소식 | 즐겨찾기 장소에 새 소식 | 새 후기/가이드가 등록됐어요 | `/places/{placeId}` | `{placeId}` |
| `FAVORITE_NEWS_PUBLISHED` | `FAVORITE` | 관심 프로젝트 뉴스 발행 | 관심 프로젝트 새 소식 | 방금 공식 뉴스가 올라왔어요 | `/info/news/{newsId}` | `{newsId}` |
| `COMMENT_CREATED` | `COMMENT` | 내 게시글에 댓글 | 내 글에 새 댓글이 달렸어요 | {actorName}: {댓글 앞 40자} | `/board/posts/{postId}` | `{postId}` |
| `COMMENT_REPLY_CREATED` | `COMMENT` | 내 댓글에 답글 | 내 댓글에 답글이 달렸어요 | {actorName}: {답글 앞 40자} | `/board/posts/{postId}` | `{postId}` |
| `SYSTEM_NOTICE` | (시스템) | 운영 공지 | 서비스 공지 안내 | 점검/정책 변경 내용을 확인하세요 | `/notifications` | `{notificationId}` |

문구 가이드:
- 제목 16~28자 권장
- 본문 28~60자 권장
- 숫자/시간/장소 등 즉시 판단 가능한 정보 우선

## 7) QA 체크리스트 (백엔드/앱 공통)
- iOS/Android 모두 foreground/background/terminated에서 알림센터 노출 확인
- 동일 이벤트 재시도 시 중복 알림 생성되지 않는지 확인 (`notificationId` 기준)
- 알림 탭 시 의도한 경로로 라우팅되는지 확인
- 카테고리 OFF 사용자에게 해당 알림이 미발행되는지 확인
- `/api/v1/notifications`와 push payload의 `notificationId`가 동일한지 확인

## 8) 확인 요청
1. 위 `eventCode`/카테고리 매핑으로 발행 가능 여부

3. 카테고리 필터링 적용 위치(큐 적재 전/후) 확정
4. 중복 방지 키를 `notificationId` 단일 키로 고정 가능한지 여부
