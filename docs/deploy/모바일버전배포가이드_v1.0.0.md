## 1. 개요

Android와 iOS 각각 독립적인 CI/CD 파이프라인을 운영합니다.

| 플랫폼 | CI/CD | 배포 대상 |
|--------|-------|-----------|
| Android | GitHub Actions | Google Play |
| iOS | Xcode Cloud | App Store Connect / TestFlight |

버전의 단일 소스는 `pubspec.yaml`의 `version` 필드입니다.  
모든 빌드는 이 값을 기준으로 버전명을 결정합니다.

---

## 2. 버전 명명 규칙

### 2.1 형식

```
MAJOR.MINOR.PATCH
```

`pubspec.yaml` 예시:

```yaml
version: 1.2.3+5
#        ↑ 버전명  ↑ 로컬 기본값(CI에서 덮어씀)
```

### 2.2 버전 증가 기준

| 구분 | 증가 조건 | 예시 |
|------|-----------|------|
| **MAJOR** | 앱 구조 대개편, 기존 데이터와 호환 불가 변경 | `0.x.x` → `1.0.0` |
| **MINOR** | 신규 기능 추가, 기존 기능 변경 | `1.2.x` → `1.3.0` |
| **PATCH** | 버그 수정, UI 소수정, 성능 개선 | `1.2.3` → `1.2.4` |

### 2.3 버전 올리는 방법

스크립트로 `pubspec.yaml`을 자동 갱신합니다:

```bash
./scripts/bump_version.sh patch   # 버그 수정
./scripts/bump_version.sh minor   # 기능 추가
./scripts/bump_version.sh major   # 대규모 변경
```

### 2.4 릴리스 태그 형식

```
v{MAJOR}.{MINOR}.{PATCH}
```

예시: `v1.3.0`, `v1.3.1`

태그 버전은 반드시 `pubspec.yaml` 버전과 일치해야 합니다.  
불일치 시 CI가 자동으로 실패 처리합니다.

## 3. 빌드 번호 정책

플랫폼마다 빌드 번호 체계가 다릅니다.

### 3.1 Android (versionCode)

GitHub Actions의 `GITHUB_RUN_NUMBER`를 사용합니다.

- 워크플로 실행마다 1씩 자동 증가
- `android-internal-distribution`과 `android-release-from-tag` 각각 독립 카운터
- Android versionCode 상한(2,100,000,000)까지 여유 충분

```
빌드 번호 = GITHUB_RUN_NUMBER
```

### 3.2 iOS (CFBundleVersion)

Xcode Cloud가 자동 관리합니다.

- Workflow 설정에서 **Automatically manage version and build number** 활성화 시 TestFlight 업로드마다 자동 증가
- Android와 독립적으로 관리되므로 번호가 다를 수 있음 — 정상

### 3.3 버전명 동기화

iOS는 Xcode Cloud 빌드 시 `ci_post_clone.sh` 안에서 아래 명령이 실행되어 `pubspec.yaml` 버전이 `project.pbxproj`에 자동 반영됩니다:

```sh
flutter build ios --release --config-only
```

## 4. 배포 트랙 구조

### 4.1 Android — Google Play 트랙

```
Internal Testing  →  Open Testing (Beta)  →  Production
      ↑                                           ↑
  main 푸시 시                              vX.Y.Z 태그 시
  자동 배포                               draft 업로드 후
                                          수동 게시
```

| 트랙 | 대상 | 배포 방식 | 상태 |
|------|------|-----------|------|
| Internal Testing | 내부 팀원 (최대 100명) | `main` 푸시 → 자동 | `completed` |
| Open Testing | 외부 베타 테스터 | 수동 승격 | 수동 |
| Production | 전체 사용자 | 태그 푸시 → draft 업로드 후 수동 게시 | `draft` → 수동 게시 |

### 4.2 iOS — TestFlight / App Store

```
TestFlight Internal  →  TestFlight External (Beta)  →  App Store
         ↑                                                  ↑
   main 푸시 시                                       vX.Y.Z 태그 또는
  Xcode Cloud 자동 빌드                              수동 제출 후 심사
```

| 단계 | 대상 | 배포 방식 |
|------|------|-----------|
| TestFlight Internal | 내부 팀원 (App Store Connect 등록자) | `main` 푸시 → Xcode Cloud 자동 |
| TestFlight External | 외부 베타 테스터 (최대 10,000명) | 수동 그룹 추가 (Apple 베타 심사 필요) |
| App Store | 전체 사용자 | 수동 제출 후 Apple 심사 통과 후 게시 |

## 5. 배포 절차

### 5.1 내부 테스트 배포 (일상적인 개발)

개발 완료 후 `main`에 머지하면 Android/iOS 모두 자동 배포됩니다.

```bash
# 1. main에 머지 (PR 또는 직접 푸시)
git push origin main

# 자동 실행:
# - Android: GitHub Actions → Google Play Internal
# - iOS: Xcode Cloud → TestFlight Internal
```

### 5.2 베타 테스트 배포 (외부 테스터)

**Android**
1. Play Console → Open Testing 트랙
2. 내부 테스트 빌드를 Open Testing으로 수동 승격
3. 테스터 초대 링크 공유

**iOS**
1. App Store Connect → TestFlight → External Testing
2. 빌드 선택 후 외부 그룹 추가
3. Apple 베타 심사 제출 (보통 1~2일 소요)

### 5.3 정식 릴리스

```bash
# 1. 버전 업데이트
./scripts/bump_version.sh minor  # 또는 patch, major

# 2. 커밋 & 푸시
git add pubspec.yaml
git commit -m "chore: bump version to 1.3.0"
git push origin main

# 3. 태그 생성 & 푸시 (Android 배포 트리거)
git tag v1.3.0
git push origin v1.3.0

# 자동 실행:
# - Android: GitHub Actions → Play Production (draft)
```

**Android 최종 게시**
- Play Console → Production 트랙 → 해당 draft 릴리스 선택 → **게시** 클릭
- 단계적 출시(%)로 리스크 최소화 권장

**iOS App Store 제출**
- App Store Connect → 해당 빌드 선택 → **심사 제출**
- Apple 심사 통과 후 수동 게시 또는 자동 게시 설정 가능

## 6. CI/CD 워크플로 구조

### 6.1 Android

| 파일 | 트리거 | 역할 |
|------|--------|------|
| `android-internal-distribution.yml` | `push → main`, `workflow_dispatch` | 분석/테스트 + Internal 배포 |
| `android-release-from-tag.yml` | `push tag v*.*.*`, `workflow_dispatch` | 분석/테스트 + Production draft 업로드 |

### 6.2 iOS

| 구성 | 트리거 | 역할 |
|------|--------|------|
| Xcode Cloud Workflow | `push → main` | 빌드 + TestFlight Internal 배포 |
| `ios/ci_scripts/ci_post_clone.sh` | Xcode Cloud 빌드 시 자동 실행 | Flutter 설치, pubspec 버전 동기화, pod install |

### 6.3 워크플로 흐름 요약

```
push → main
  ├── GitHub Actions: flutter analyze → test → build AAB → Play Internal
  └── Xcode Cloud: ci_post_clone.sh → Archive → TestFlight Internal

push tag vX.Y.Z
  └── GitHub Actions: flutter analyze → test → build AAB → Play Production (draft)
      (iOS는 수동으로 App Store Connect에서 심사 제출)
```

## 7. 플랫폼별 시크릿 설정

### 7.1 GitHub Secrets (Android용)

GitHub 저장소 → Settings → Secrets and variables → Actions 에서 등록

| Secret 이름 | 설명 | 생성 방법 |
|-------------|------|-----------|
| `ANDROID_UPLOAD_KEYSTORE_BASE64` | 서명 keystore (base64) | `base64 -i android/app/upload-keystore.jks \| pbcopy` |
| `ANDROID_KEYSTORE_PASSWORD` | keystore 비밀번호 | `android/key.properties` 참조 |
| `ANDROID_KEY_PASSWORD` | key 비밀번호 | `android/key.properties` 참조 |
| `ANDROID_KEY_ALIAS` | key alias | `android/key.properties` 참조 |
| `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON` | Play Developer API 서비스 계정 JSON | Google Cloud Console에서 생성 |

### 7.2 Xcode Cloud Environment Variables (iOS용)

Xcode → ⌘9 → Cloud → Workflow 편집 → Environment 에서 등록

현재 `GoogleService-Info.plist`가 저장소에 커밋되어 있으므로 별도 환경변수 불필요.  
향후 plist를 저장소에서 제거할 경우 아래 중 하나 설정:

| 변수 이름 | 설명 |
|-----------|------|
| `GOOGLE_SERVICE_INFO_PLIST_B64` | plist 파일 base64 인코딩 값 |

생성 방법:
```bash
base64 -i ios/Runner/GoogleService-Info.plist | tr -d '\n'
```

## 8. 문제 해결

### 8.1 `Tag/Input version does not match pubspec version`

- **원인**: 태그 버전과 `pubspec.yaml` 버전 불일치
- **조치**: `pubspec.yaml` 버전을 태그와 동일하게 수정 후 태그 재생성

```bash
git tag -d v1.3.0
git push origin :refs/tags/v1.3.0
# pubspec.yaml 수정 후
git tag v1.3.0
git push origin v1.3.0
```

### 8.2 Play 업로드 권한 오류

- **원인**: 서비스 계정 권한 부족 또는 Google Play Android Developer API 미활성화
- **조치**:
  1. Google Cloud Console → API 및 서비스 → `Google Play Android Developer API` 활성화
  2. Play Console → 사용자 및 권한 → 서비스 계정 → 릴리스 관리 권한 부여

### 8.3 Xcode Cloud `ci_post_clone.sh` 실패

- **원인**: `GoogleService-Info.plist` 파일이 저장소에 없음
- **조치**: 파일을 저장소에 커밋

```bash
git add -f ios/Runner/GoogleService-Info.plist
git commit -m "chore: add GoogleService-Info.plist"
git push
```

### 8.4 iOS 빌드 버전이 pubspec과 다름

- **원인**: `ci_post_clone.sh`에서 `flutter build ios --release --config-only` 미실행
- **조치**: `ios/ci_scripts/ci_post_clone.sh`에 해당 명령이 포함되어 있는지 확인

### 8.5 TestFlight External 심사 지연

- Apple 베타 심사는 보통 1~2일 소요
- 동일 빌드를 Internal에서 먼저 검증 후 External 제출 권장
